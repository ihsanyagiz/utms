# Classes Directory — Coding Rules

**Location**: `app/classes/`  
**Purpose**: Business logic layer with role-specific operations  

## Structure

```
classes/
  users/
    base.py       # Abstract User(ABC) — do not confuse with UserORM
    admin.py      # Admin role: user mgmt, config view, DB backup/restore
    applicant.py  # Applicant role: submit, resubmit, cancel
    dean.py       # DeanOffice: review, return to applicant/OIDB/YDYO, forward to YGK
    oidb.py       # Oidb: document verification
    ydyo.py       # Ydyo: prep school marking
    ygk.py        # YGK: approval
    __init__.py   # Exports all role classes
  auth.py         # AuthClass (credentials, tokens)
  entities/       # Domain entities (Application, Document) — NOT ORM models
  interfaces.py   # IReviewer, IDocumentHandler abstract bases
```

**Naming note**: `classes/users/base.py::User` is the abstract business base class. `models/user.py::UserORM` is the SQLAlchemy ORM model. They are different things. Never import UserORM into business classes directly.

## EXPLICIT CODING RULES

### ✅ DO:
1. **Use repository interfaces** — `get_application_repository()` for application data, `get_user_repository()` for user data (Admin only)
2. **Return OperationResult** — every public method returns `OperationResult` with success/error_code/data/error_msg
3. **Use specific ErrorCodes** — `ErrorCode.VALIDATION_FAILED`, `NOT_FOUND`, etc. (never bare strings)
4. **Handle exceptions** — catch and convert to OperationResult with appropriate ErrorCode
5. **Instantiate per request** — classes receive `(user_id, email, role)` from the route and are short-lived

### ❌ DO NOT:
1. **Direct db.session** — never import `db` or call `db.session.*` directly
2. **Direct model queries** — never call `ApplicationORM.query.filter(...)` directly; use repository
3. **Import UserORM in business classes** — only repositories import ORM models
4. **Bare bool returns** — never return `True/False`; wrap in `OperationResult`
5. **Form parsing** — move form handling to routes/services, not here
6. **Manually set `last_edited_by_id` / `last_edited_at`** — the `before_flush` listener handles it automatically

## Pattern: Correct Repository Usage

```python
def forward_to_ygk(self, application_id: int, faculty: str) -> OperationResult:
    app_repo = get_application_repository()
    
    result = app_repo.find_by_id(application_id)
    if not result.success:
        return result
    if result.data is None:
        return OperationResult.error(ErrorCode.APPLICATION_NOT_FOUND, "Not found.")
    
    app = result.data
    if app.status != "forwarded_to_dean":
        return OperationResult.error(
            ErrorCode.INVALID_STATUS_TRANSITION,
            f"Cannot forward from status '{app.status}'."
        )
    
    app.forwarded_faculty = faculty
    app.status = "forwarded_to_ygk"
    return app_repo.update(app)   # auto-fingerprinted by before_flush listener
```

## DeanOffice — Return and Forward

`DeanOffice` is the soft checkpoint before YGK. It can forward or send applications back.

```python
# Return to a previous stage with a reason note
result = DeanOffice(user_id, email, "dean").return_application(
    application_id=42,
    target="oidb",          # "applicant" | "oidb" | "ydyo"
    notes="Please re-verify the transcript."
)
# → sets app.dean_notes, transitions status: applicant→"returned", oidb→"submitted", ydyo→"forwarded_to_ydyo"

# Forward to YGK
result = DeanOffice(user_id, email, "dean").forward_to_ygk(application_id=42, faculty="Faculty of Engineering")
```

`dean_notes` is displayed as a yellow warning banner in the OIDB panel, YDYO panel, and the applicant's track view so each receiving role understands why the application was returned.

## Document Checker Injection (Oidb and Ydyo)

`Oidb` and `Ydyo` both have an `auto_check_and_forward_all(checker=None)` method. When `checker` is `None` it defaults to `MockDocumentChecker`. Pass a real `IDocumentChecker` implementation to swap in production behaviour:

```python
from app.services.document_checker import IDocumentChecker

class MyChecker(IDocumentChecker):
    def check_application(self, application) -> DocumentCheckResult:
        # real logic here
        ...

result = Oidb(user_id, email, "oidb").auto_check_and_forward_all(MyChecker())
```

The method stores `doc_checker_status = "auto_checked"` and `doc_checker_errors` (JSON) on every processed application, then forwards those that passed and leaves the rest in place with their errors visible in the UI.

**`Oidb.auto_check_and_forward_all`** — processes applications in `"submitted"` status → forwards passing ones to `"forwarded_to_ydyo"`.  
**`Ydyo.auto_check_and_forward_all`** — processes `"forwarded_to_ydyo"` → forwards passing ones to `"forwarded_to_dean"` with `prep_school_status = "eligible"`.

## Admin Class — Extra Capabilities

`Admin` is the only role class that also uses `get_user_repository()` (for user listing and role changes) and `get_database_interface()` (for backup/restore).

```python
from ...database import get_application_repository, get_database_interface, get_user_repository
```

All other role classes only use `get_application_repository()`.

## Anti-Pattern: What NOT to Do

```python
# ❌ WRONG — direct db.session
from app import db
app = db.session.query(ApplicationORM).filter(...).first()
db.session.add(something)
db.session.commit()

# ❌ WRONG — direct ORM model import
from ...models import ApplicationORM
ApplicationORM.query.filter_by(...)  # belongs in repository

# ❌ WRONG — bare bool return
def validate_form(self, data: dict) -> bool:
    return "email" in data

# ❌ WRONG — manual fingerprint
app.last_edited_by_id = self.user_id   # the listener does this automatically
```
