# Services Directory — Coding Rules

**Location**: `app/services/`  
**Purpose**: Business logic orchestration layer (mediator between routes and repositories)

## Structure
- `__init__.py` — export service getter functions
- `application_service.py` — submit, resubmit, cancel, get
- `auth_service.py` — login, register, validate_credentials
- `user_service.py` — user-level helpers
- `base_service.py` — BaseService with `ok()`, `error()`, `validate_required_fields()`
- `document_checker.py` — `IDocumentChecker` interface + `MockDocumentChecker` implementation

## Document Checker (`document_checker.py`)

This module defines the **pluggable document verification interface**.

```python
class IDocumentChecker(ABC):
    def check_application(self, application) -> DocumentCheckResult: ...

class MockDocumentChecker(IDocumentChecker):
    # Checks: required slots 1-3 present, GPA >= 2.50 (minimum transfer GPA), id_number is numeric
    ...
```

`DocumentCheckResult` has `.passed: bool` and `.errors: list[DocumentCheckError]`.  
`DocumentCheckError` has `.type` ("document"/"field"), `.reason`, `.slot`, `.field_name`.

**To swap in a real checker**: implement `IDocumentChecker`, instantiate it, and pass it to `Oidb.auto_check_and_forward_all(my_checker)` or `Ydyo.auto_check_and_forward_all(my_checker)`. No other code changes needed.

**Do not import** `MockDocumentChecker` in business code outside of the default argument fallback in `oidb.py`/`ydyo.py`. Routes do not touch the checker directly.

## EXPLICIT CODING RULES

### ✅ DO:
1. **Orchestrate business logic** - Combine multiple repository calls for complex operations
2. **Return OperationResult** - All service methods return OperationResult with error codes
3. **Use repositories internally** - Call `get_application_repository()` inside services
4. **Handle form data** - Services accept request.form and parse/validate it
5. **Return user-friendly messages** - error_msg should be readable for toasts
6. **Keep methods focused** - One operation per method; avoid god methods

### ❌ DO NOT:
1. **Use db.session** - Only repositories touch db.session; services use repositories
2. **Accept Flask request object** - Services receive form/files as parameters, not request
3. **Mix HTTP concerns** - No session access, no redirect logic; let routes handle that
4. **Direct model instantiation** - Don't import models; use repositories
5. **Return bare types** - Always wrap in OperationResult, never bare dict/list/bool

## Pattern: Correct Service Implementation

```python
# app/services/application_service.py
from app.database import get_application_repository
from app.classes import Applicant
from app.core import OperationResult, ErrorCode

class ApplicationService:
    def __init__(self, user_id: int):
        self.user_id = user_id
        self.user = Applicant(user_id=user_id, email="...", role="applicant")
    
    def submit_application(
        self, 
        form_data: dict, 
        files: dict,
        upload_root: str
    ) -> OperationResult:
        """Submit new application with documents."""
        
        # 1. Validate form
        if not form_data.get("email"):
            return OperationResult.error(
                ErrorCode.VALIDATION_FAILED,
                "Email is required"
            )
        
        # 2. Use class to handle submission
        result = self.user.submit(form_data, files, upload_root, {})
        return result
    
    def get_pending_applications(self) -> OperationResult:
        """Get all pending applications for current user."""
        app_repo = get_application_repository()
        return app_repo.find_by_applicant_id(self.user_id)
```

## Anti-Pattern: What NOT to Do

```python
# ❌ WRONG - Using db.session
def submit_application(self, form_data):
    from app import db
    from app.models import Application
    app = Application(**form_data)
    db.session.add(app)
    db.session.commit()
    return True

# ❌ WRONG - Accepting Flask request
def process_files(self, request):
    for file in request.files:
        # ... complex logic ...

# ❌ WRONG - No OperationResult
def validate_form(self, data: dict) -> bool:
    return "email" in data and len(data["email"]) > 0

# ❌ WRONG - Mixing concerns
def submit_and_redirect(self, form):
    result = self.submit(form)
    if result.success:
        return redirect(url_for("dashboard"))  # HTTP concern!
```

## Service Getter Pattern

Routes use getter functions to get services:

```python
# app/services/__init__.py
def get_applicant_service(user_id: int) -> ApplicationService:
    return ApplicationService(user_id)

def get_dean_service(user_id: int) -> DeanService:
    return DeanService(user_id)
```

Routes use it:
```python
# app/routes/applicant.py
from app.services import get_applicant_service

@applicant_api_bp.route("/submit", methods=["POST"])
def submit_application():
    service = get_applicant_service(session["user_id"])
    result = service.submit_application(request.form, request.files)
    if result.success:
        flash(result.data.get("message"), "success")
    else:
        flash(result.error_msg, "error")
    return redirect(...)
```

## Error Messages for Toasts

Services should provide user-friendly messages in `OperationResult.error_msg`:

```python
# ✅ GOOD - User-friendly
return OperationResult.error(
    ErrorCode.VALIDATION_FAILED,
    "Email address is already registered"
)

# ❌ BAD - Technical jargon
return OperationResult.error(
    ErrorCode.DUPLICATE,
    "IntegrityError: UNIQUE constraint failed: user.email"
)
```
